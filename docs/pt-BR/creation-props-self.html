Creation, Props & Self

The Record API is in feature-freeze. For the newest features and better support going forward, please consider migrating to the new function components.

The documentation assumes relative familiarity with ReactJS.

ReasonReact doesn't use/need classes. The component creation API gives you a plain record, whose fields (like render) you can override.

Um template de component é criado através de ReasonReact.statelessComponent("NomeDoComponente"). The string being passed is for debugging purposes (the equivalent of ReactJS' displayName).

As an example, here's the file Greeting.re:

let component = ReasonReact.statelessComponent("Greeting");

In ReactJS, you'd create a component class and call it through JSX which transforms into React.createElement(myClass, {prop1: 'hello'}) under the hood. In ReasonReact, instead of passing the whole "class" (aka component template) into a hypothetical ReasonReact.createElement function, you'd instead declare a make function:

/* still in Greeting.re */
let component = ReasonReact.statelessComponent("Greeting");

let make = (~name, _children) => {
  ...component, /* spread the template's other defaults into here  */
  render: _self => <div> {ReasonReact.string(name)} </div>
};

The make function is what's called by ReasonReact's JSX, described later. For now, the JSX-less way of calling & rendering a component is:

ReasonReact.element(Greeting.make(~name="John", [||])) /* the `make` function in the module `Greeting` */
/* equivalent to <Greeting name="John" /> */

make asks you to return the component record created above. You'd override a few fields, such as the familiar render, initialState, didMount, etc., all described later.

Nota: Não insira em linha let componente dentro do corpo da função make como a seguir!

let make = _children => {
  ...(ReasonReact.statelessComponent("Greeting")),
  render: self => blabla
}

Sendo o make chamado a cada chamada JSX, você estaria criando um componente novo a cada chamada.

Props

Props são como argumentos rotulados da função make, vista acima. They can also be optional and/or have defaults, e.g. let make = (~name, ~age=?, ~className="box", _children) => /* ... */.

A última prop deve ser children. Se você não usá-la, simplesmente a ignore a chamando de _ ou _children. Nomes começados por sublinhado não ativam as mensagens de aviso de não usado no compilador.

O nome da prop não pode ser ref nem key. Pois são reservadas, que nem no ReactJS.

Seguindo aquele exemplo, você chamaria aquele componente em outro arquivo através de <Foo nome="Reason" />. className, se omitida, é associada para "box". idade é associada à None. Se você gostaria de passar explicitamente idade, simplesmente o faça: <Foo nome="Reason" idade=20 />.

Dica Legal para dar Forward nas Props

Algumas vezes em ReactJS, você recebe uma prop de um pai que você gostaria de passar diretamente para um filho:

<Foo nome="Reason" age={this.props.age} />

Isso pode originar bugs, pois this.props.idade pode ser mudada acidentalmente para um número nulável enquanto Foo não o espera que seja e vice-versa; poderia ser nulável antes e agora não e Foo acaba ficando com uma checagem de nulo inútil em algum lugar do render.

In Reason, if you want to explicitly pass an optional myAge (whose type is option(int), aka None | Some(int)), the following wouldn't work:

<Foo name="Reason" age=myAge />

Because the age label expects a normal int when you do call Foo with it, not an option(int)! Naively, you'd be forced to solve this like so:

switch (myAge) {
| None => <Foo name="Reason" />
| Some(nonOptionalAge) => <Foo name="Reason" age=nonOptionalAge />
}

Trabalhoso. Felizmente, temos um jeito melho de passar explicitamente um valor opcional:

<Foo name="Reason" age=?myAge />

It says "I understand that myAge is optional and that I should either use the label age and pass an int, or not use the label at all. But I'd like to forward an optional value explicitly to avoid the verbose switch".

This isn't a JSX trick we've made up; it's just a language feature! See the section on "Explicitly Passed Optional" in the Reason docs.

self

You might have seen the render: (self) => ... part in make. The concept of JavaScript this doesn't exist in ReasonReact (but can exist in Reason, since it has an optional object system); the this equivalent is called self. It's a record that contains state, handle and send, which we pass around to the lifecycle events, render and a few others, when they need the bag of information. These concepts will be explained later on.
