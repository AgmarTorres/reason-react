Callback Handlers

Esta sección describe como se traduce en ReasonReact el patrón <div onClick={this.handleClick} /> de ReactJS.

Callback Sin Actualización del State

Aquí tenemos dos escenarios.

No Leyendo de self

Reminder: self is ReasonReact's this. It's a record that contains things like state, send and others.

Si solamente estás pasando una prop tipo callback a tu hijo, harías exactamente lo mismo que habrías hecho en ReactJs:

let component = /* ... */;

let make = (~name, ~onClick, _children) => {
  ...component,
  render: (self) => <button onClick=onClick />
};

Sin novedad por aquí. Como JSX en Reason implementa abreviación sintáctica, ese button se transformaría en <button onClick />.

Del mismo modo, para preprocesar un valor antes de mandarlo de vuelta al consumidor del componente:

let component = /* ... */;

let make = (~name, ~onClick, _children) => {
  let click = (event) => onClick(name); /* pass the name string up to the owner */
  {
    ...component,
    render: (self) => <button onClick=click />
  }
};

Accediendo a self

To access state, send and the other items in self from a callback, you need to wrap the callback in an extra layer called self.handle:

let component = /* ... */;

let make = (~name, ~onClick, _children) => {
  let click = (event, self) => {
    onClick(event);
    Js.log(self.state);
  };
  {
    ...component,
    initialState: /* ... */,
    render: (self) => <button onClick={self.handle(click)} />
  }
};

Observa cómo tu click callback recibe ahora el argumento extra self. Formalmente, self.handle espera un callback que:

  • acepta los parámetros que normalmente pasarías a e.g. handleClick,
  • más el parámetro self,
  • devuelve "nada" (véase, (), véase, unit).

Observación 2: en algunas ocasiones querrás pasar handle a alguna función de apoyo. Es este caso, pasa self directamente y anótalo. Esto evita complicar demasiado el comportamiento del tipo self. Véase Record Field send/handle no encontrado.

Explicación

En realidad, self.handle es una función normal y corriente que acepta dos parámetros. El primero es el callback en cuestión, y el segundo los datos que queremos que el callback reciba.

¿Entiendes? Debido al currying en Reason, generalmente sólo te pedimos pasar un parámetro. Esto devuelve una nueva función que toma el segundo parámetro y ejecuta la función. El segundo parámetro en este caso está siendo pasado por el "llamador", es decir, el componente que estás renderizando.

Callback Recibiendo Múltiples Parámetros

Algunas veces, el componente que estamos llamando proviene de Javascript, (utilizando la interoperabilidad de ReasonReact<->ReactJS ), y su prop de callback pide que pases una función de callback que acepte más de un argumento. En ReactJS, sería algo así:

handleSubmit: function(username, password, event) {
  this.setState(...)
}
...
<MyForm onUserClickedSubmit={this.handleSubmit} />

No puedes escribir esta función handleSubmit en ReasonReact, ya que handle espera una función que recibe sólo un parámetro. Así es como lo solucionarías:

let handleSubmitEscapeHatch = (username, password, event) =>
  self.handle(
    (tupleOfThreeItems, self) => doSomething(tupleOfThreeItems, self),
    (username, password, event),
  );
...
<MyForm onUserClickedSubmit=(handleSubmitEscapeHatch) />

Básicamente, escribirías una función de callback normal que:

  • toma todos los argumentos de la prop de callback el componente JS,
  • los empaqueta en una tupla y llama a self.handle,
  • pasa a handle una función que espera un solo argumento
  • acaba llamando a self.handle pasándole la tupla directamente.
